# 1. Самое главное

В IPaC нельзя путать движение по процессу с получением результата.

```text
Хороший процесс должен быстро давать проверяемый комплексный результат.
Плохой процесс бесконечно улучшает отдельные детали.
```

# 2. Главная формула

```text
Сначала видимый комплекс.
Потом реализуемость.
Потом целесообразность.
Потом полировка.
```

# 3. Что такое видимый комплекс

Это не идеальный документ.

Это:

```text
1. Артефакт, который можно открыть.
2. Статус candidate / not_canon.
3. Понятная проблема.
4. Кандидатная формула.
5. Карта компонентов.
6. Схема процесса.
7. Риски.
8. Следующий практический шаг.
```

# 4. Ошибка, которую нельзя повторять

```text
Не надо пять раз улучшать один раздел,
если ещё не доказано, что весь контур вообще нужен и реализуем.
```

# 5. Роль Supervisor

Supervisor — не начальник бюрократии.

Supervisor — диспетчер результата.

Он должен удерживать:

```text
scope;
status;
visible output;
feasibility;
expediency;
Codex handoff;
Human Approval boundary.
```

# 6. Роль Codex

Codex — практик и технический оппонент.

Он должен проверять:

```text
можно ли это собрать;
какие инструменты нужны;
что блокирует MVP;
что автоматизировать первым.
```

Codex не должен:

```text
канонизировать;
самовольничать;
делать commit без approval;
переписывать архитектурное ядро.
```

# 7. Как действовать стажёру

```text
1. Найди текущий visible result.
2. Найди статус: candidate / review / decision / canon.
3. Найди ближайший feasibility test.
4. Найди ближайший expediency test.
5. Не полируй до прохождения этих ворот.
6. Всегда требуй check file после файлового действия.
```

# 8. Мини-памятка

```text
Result first.
Then proof.
Then decision.
Then polish.
```
