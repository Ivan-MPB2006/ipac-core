# 1. Purpose

Этот документ фиксирует стратегическую интерпретацию DR-002 и обсуждения после отчёта.

Главная задача:

```text
не раздуть из мухи слона,
но и не обращаться со слоном как с мухой.
```

Это означает:

```text
IPaC не должен преждевременно объявлять себя новой инженерной дисциплиной.

Но IPaC также не должен преждевременно сжиматься до ближайших малых соседей:
Context Engineering,
Prompt Engineering,
Docs-as-Code,
Obsidian + Git workflow,
Knowledge Management или RAG pipeline.
```

---

# 2. Strategic Thesis

```text
Минимально нам достаточно IPaC как framework.

Но исследование нужно,
чтобы понять,
не имеем ли мы дело с более крупным инженерным сдвигом,
который нельзя сводить к частным AI-era практикам.
```

Иначе говоря:

```text
Минимум:
IPaC = human-AI framework для дисциплинированного производства,
проверки, сопровождения и переиспользования knowledge artifacts.

Максимум:
IPaC = заявка на Engineering of Information Production,
где software engineering, context engineering, documentation engineering,
knowledge engineering и decision-support artifacts становятся частными слоями.
```

Максимум остаётся гипотезой.

---

# 3. What DR-002 Clarified

DR-002 показал, что Context Engineering на текущем уровне лучше понимать как:

```text
emerging AI-era practice cluster
```

а не как зрелую инженерную дисциплину.

Он возник вокруг реальной боли:

```text
как собрать, отформатировать, ограничить,
доставить и обновить context,
который модель / агент должен видеть сейчас.
```

При этом сама проблема старше термина и имеет корни в:

- pragmatics;
- situated cognition;
- distributed cognition;
- human factors;
- cognitive systems engineering;
- context-aware computing;
- knowledge engineering;
- information retrieval;
- prompt engineering;
- RAG;
- memory and agent tooling.

Следовательно:

```text
свежесть термина не делает объект новым;
популярность практики не делает её зрелой дисциплиной.
```

# 4. The “Engineering” Label Problem

Ключевой вывод:

```text
Не всякое “Engineering” является инженерной дисциплиной.
```

Context Engineering, Prompt Engineering, Agent Engineering, RAG Engineering и подобные названия могут обозначать:

```text
реальную практическую боль;
набор полезных приёмов;
роль;
toolchain pattern;
маркетинговый термин;
или emerging practice cluster.
```

Но само слово “engineering” ещё не доказывает наличие:

- собственного объекта;
- методов;
- lifecycle;
- validation culture;
- стандартов;
- институциональной базы;
- профессиональной ответственности;
- понятных failure modes.

Примерная формула:

```text
Если любую toolchain-боль назвать engineering,
то открывается ящик Пандоры:
можно объявить отдельной инженерией
решение задач Big Data на языке R,
создание RAG-pipeline,
настройку prompt,
или сборку агентной памяти.
```

Поэтому для IPaC введено правило:

```text
Сначала определить статус соседней области.
Потом определять границу.
```

# 5. Context Engineering as Part of Broader Cognitive Work

Сильная интерпретация:

```text
Context Engineering может быть не самостоятельной дисциплиной,
а AI-era частным слоем более широкой когнитивно-инженерной задачи.
```

Более широкая задача:

```text
обеспечить субъекту действия — человеку, AI-модели, агенту или команде —
максимально релевантный контекст
для понимания,
мышления,
решения,
действия
и проверки.
```

Эта задача давно существовала в:

- Cognitive Engineering;
- Human Factors;
- HCI;
- Decision Support;
- Situation Awareness;
- Knowledge Engineering;
- Information Architecture.

LLM сделал эту задачу более видимой, острой и технически операциональной.

# 6. Context vs Knowledge

DR-002 усилил ключевое различение:

```text
Контекст — это то, что нужно модели / агенту / человеку сейчас
для конкретного действия или интерпретации.

Знание — это то, что должно сохраняться,
проверяться,
получать обратную связь,
развиваться,
переиспользоваться
и выдерживать время.
```

Отсюда:

```text
Context Engineering оптимизирует момент взаимодействия с моделью.

IPaC проектирует жизненный цикл знания во времени.
```

Эта формула остаётся research-supported hypothesis, не field validation.

---

# 7. Feedback / Reality Coupling as Key Boundary

Самая сильная граница после DR-002:

```text
Context Engineering улучшает следующее действие агента.

IPaC пытается изменить и развить состояние знания,
которое должно пройти review,
decision,
trace,
controlled rebuild
и пережить время.
```

То есть IPaC не только удерживает знание.

Более точная формула:

```text
IPaC — это контур развития знания через обратную связь с реальностью.
```

Эта формула должна быть проверена в следующих research / field validation cycles.

# 8. Relation to Software Engineering

Важная стратегическая гипотеза:

```text
Software Engineering является не внешним аналогом,
а потенциально частным слоем более широкого Engineering of Information Production.
```

Почему:

```text
Software Engineering работает с исполняемыми артефактами.

IPaC рассматривает более широкий класс:
смыслонесущие информационные артефакты,
включая исследовательские,
управленческие,
коммуникационные,
аналитические,
документационные,
контекстные,
решенческие
и программные артефакты.
```

Код — частный случай информационного артефакта с особым свойством:

```text
он исполняется машиной.
```

Но далеко не все информационные артефакты исполняются как код.

# 9. Software Engineering Under AI Pressure

Рабочая гипотеза:

```text
AI требует пересборки software engineering.
```

Старый контур software engineering во многом строился вокруг предположения:

```text
человек пишет код;
машина исполняет;
инженер проверяет;
pipeline собирает.
```

AI вводит новый контур:

```text
человек задаёт рамку;
AI генерирует;
агент меняет файлы;
context влияет на поведение;
knowledge и code смешиваются;
проверка становится процессной,
вероятностной
и многоуровневой.
```

Следовательно:

```text
AI нельзя просто примысливать к старым процессам.
AI требует перепроектирования самого информационного производства.
```

Важно:

```text
Ссылки на кризисы внедрения AI,
MIT / McKinsey / corporate AI losses
должны быть отдельно проверены source scan,
прежде чем использоваться как strong evidence.
```

# 10. Dotcom Analogy and Warning

Аналогия:

```text
До индустриальной зрелости software engineering
многое в программировании оставалось ремесленным производством.

AI может создать новый кризис,
если его будут механически прикручивать к старым процессам,
не создавая нового инженерного контура.
```

Не утверждать:

```text
новый dotcom crisis неизбежен.
```

Безопасная формулировка:

```text
AI-era adoption risks may grow if organizations keep treating AI
as a tool add-on rather than redesigning the information-production system.
```

Русская версия:

```text
Риски внедрения AI могут расти,
если организации будут прикручивать AI к старым процессам,
а не перепроектировать сам контур информационного производства.
```

# 11. What to Capture in DR-002 Review

DR-002 Research Review должен включить следующие тезисы:

1. Context Engineering нельзя принимать как зрелую дисциплину только из-за слова “Engineering”.
2. Context Engineering сейчас выглядит как emerging practice cluster вокруг model-facing context.
3. Его функции частично могут быть описаны как слой cognitive engineering / human-AI systems.
4. IPaC не должен обесценивать Context Engineering: оттуда нужно наследовать context assembly, memory, tools, evals, observability, safety.
5. IPaC не сводится к Context Engineering, если сохраняется объект knowledge lifecycle:
   `feedback → evidence → review → decision → controlled rebuild`.
6. Software Engineering itself is under AI pressure and may require rethinking.
7. IPaC maximum hypothesis should be framed as early theoretical work toward Engineering of Information Production, not as a proven new discipline.

# 12. Claims to Weaken

| Claim | Why Weaken | Safer Wording |
|---|---|---|
| Context Engineering is a mature engineering discipline | Evidence currently points to emerging practice cluster. | Context Engineering currently appears to be an emerging AI-era practice cluster. |
| Prompt Engineering / Context Engineering are full engineering disciplines | They may be toolchain practices or role labels. | They are useful practices whose discipline status requires testing. |
| IPaC already proves Engineering of Information Production | No field validation. | IPaC is a candidate framework and research program toward Engineering of Information Production. |
| Software Engineering is sufficient for AI-era work | AI changes code, context, knowledge and validation loops. | Software Engineering may need substantial AI-era redesign. |
| Context Pack is IPaC-unique | Strong overlap with CE patterns. | IPaC context pack is a local IPaC implementation of model-facing context discipline. |

# 13. Claims to Strengthen

| Claim | Why Strengthen | Safe Wording |
|---|---|---|
| IPaC is broader than model context delivery | DR-002 shows CE focuses on current model/agent context. | IPaC appears broader at the knowledge lifecycle and governance level. |
| Context Engineering is useful but partial | It handles model-facing context well. | CE is a valuable layer inside broader human-AI knowledge work. |
| IPaC needs engineering of retention and development | IPaC includes feedback, review, decision and controlled rebuild. | IPaC investigates how knowledge can be retained, checked, developed and reused over time. |
