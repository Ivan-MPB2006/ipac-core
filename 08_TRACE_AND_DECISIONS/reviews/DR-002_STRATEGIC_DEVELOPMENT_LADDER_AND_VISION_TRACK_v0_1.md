# 1. Purpose

Этот документ фиксирует стратегическую лестницу развития IPaC:

```text
прикладная SaaS-система
→ framework для создания таких систем
→ практическая апробация
→ теоретическое обобщение
→ теоретико-практическое движение
```

Документ также фиксирует отдельный визионерский трек:

```text
написать vision paper уже сейчас,
не объявляя доказанную дисциплину,
но показывая направление AI-era engineering reset.
```

---

# 2. Strategic Principle

```text
Нам не нужно искусственно раздувать IPaC до новой дисциплины раньше времени.

Но нам также нельзя преждевременно сжимать IPaC до частных практик:
Context Engineering,
Prompt Engineering,
Docs-as-Code,
Obsidian + Git workflow,
RAG pipeline,
Knowledge Management.
```

Рабочая позиция:

```text
Минимум должен быть реализуемым.
Максимум должен быть исследуемым.
Визия должна быть сформулирована уже сейчас.
```

# 3. Program Minimum — Applied SaaS System

## 3.1 Definition

Программа минимум:

```text
создать прикладную систему
и эксплуатировать её в SaaS-режиме.
```

Это самый защищённый путь, потому что он не требует немедленного признания новой дисциплины.

## 3.2 Object

Объект программы минимум:

```text
практическая система,
которая помогает производить, структурировать, проверять,
удерживать, обновлять и переиспользовать информационные продукты
в human-AI контуре.
```

## 3.3 Why This Is Defensible

Даже если IPaC пока остаётся только framework, прикладная SaaS-система может быть полезна.

Никто не мешает:

- создать продукт;
- проверить workflow;
- измерить value;
- отработать UX;
- построить operating loop;
- собрать field evidence;
- накопить реальные кейсы.

## 3.4 Success Criteria

| Criterion | Meaning |
|---|---|
| Working system | система реально используется |
| Repeatable workflow | есть повторяемый процесс |
| User value | пользователи получают пользу |
| Retention | knowledge artifacts не теряются |
| Feedback loop | feedback влияет на улучшения |
| Git / trace / audit | изменения прослеживаются |
| Reuse | артефакты переиспользуются |
| SaaS viability | возможна эксплуатация как сервиса |

# 4. Step Above — Framework for Building Such Systems

## 4.1 Definition

Следующий уровень:

```text
создать framework для проектирования и развёртывания подобных систем.
```

То есть не только один продукт, а набор принципов, шаблонов, процессов, ролей, артефактов и инструментов.

## 4.2 Object

Объект framework-уровня:

```text
метод и архитектурная схема,
по которой можно создавать разные IPaC-like системы
для разных предметных областей и типов информационных продуктов.
```

## 4.3 Framework Components

Framework должен включать:

- artifact lifecycle;
- canon / trace / review / decision model;
- input-pack generation;
- source quality gates;
- claims ledger;
- field validation split;
- context / knowledge distinction;
- human-AI role boundaries;
- Codex / action layer task pack rules;
- deployment patterns;
- governance patterns;
- feedback and reality coupling.

## 4.4 Success Criteria

| Criterion | Meaning |
|---|---|
| Replicability | framework можно применить в другой области |
| Configurability | можно адаптировать под разные типы knowledge work |
| Teachability | его можно объяснить и передать |
| Templates | есть шаблоны файлов, prompts, task packs |
| Tooling | есть scripts / builders / validators |
| Governance | есть правила изменения canon / claims / artifacts |
| Case library | есть набор практических кейсов |

# 5. Program Maximum — Theoretical-Practical Movement

## 5.1 Definition

Программа максимум:

```text
на базе практических апробаций
сформировать теоретико-практическое движение
по Engineering of Information Production.
```

Это не стартовая претензия, а верхний уровень после накопления practical evidence.

## 5.2 Required Preconditions

Программа максимум допустима только после:

- working SaaS/product evidence;
- repeated use cases;
- field validation;
- external users;
- documented failures;
- refined methodology;
- comparison with neighboring fields;
- claims ledger discipline;
- source-quality-based research;
- theoretical generalization.

## 5.3 What Movement Means

Под движением понимается не маркетинговый бренд, а:

```text
сообщество практики,
теоретические основания,
прикладные методики,
инструменты,
кейсы,
образовательные материалы,
исследовательские программы,
полевые проверки.
```

## 5.4 Guardrail

```text
Нельзя объявлять движение до практической апробации.

Но можно проектировать траекторию,
которая к нему потенциально ведёт.
```

# 6. Vision Track — Write the Vision Now

До SaaS, framework и движения можно уже сейчас написать vision paper.

## 6.1 Why Vision Is Allowed Now

Vision не требует утверждать:

```text
IPaC уже доказал новую дисциплину.
```

Vision может утверждать:

```text
мы видим системный разрыв;
мы видим направление движения;
мы предлагаем framework / research program / prototype path;
мы показываем, как надо мыслить иначе.
```

## 6.2 Vision Core Message

```text
AI нельзя просто примысливать к старым процессам.

AI требует перепроектирования самого информационного производства.
```

Или:

```text
Проблема не только в моделях.
Проблема в отсутствии нового инженерного контура
для human-AI information production.
```

## 6.3 External Motivation

Vision paper может опираться на современные наблюдения:

```text
enterprise AI adoption is widespread,
but scaling and measurable value remain uneven;
many AI pilots fail not because models are weak,
but because workflows, governance, integration and learning loops
are not redesigned.
```

Требование:

```text
все внешние claims должны быть подтверждены source scan
перед публикацией.
```

## 6.4 Vision Paper Candidate Title

```text
From AI Tools to Information Production Engineering:
Why Human-AI Systems Need a New Operating Contour
```

Русский вариант:

```text
От AI-инструментов к инженерии информационного производства:
почему human-AI системам нужен новый операционный контур
```

# 7. Strategic Ladder

| Level | Name | Claim Strength | Output | Validation Needed |
|---|---|---|---|---|
| L1 | Applied SaaS System | practical product claim | SaaS / working system | user adoption, value, retention |
| L2 | Framework | methodological claim | framework / templates / tooling | repeatable use across cases |
| L3 | Research Program | research claim | research sprints / reviews / decisions | source quality, boundary clarity |
| L4 | Vision Paper | strategic claim | article / white paper / manifesto | source-backed argument |
| L5 | Theoretical-Practical Movement | discipline-building claim | community, cases, methods, education | field validation and adoption |
| L6 | Engineering of Information Production | maximum hypothesis | theoretical-practical domain | long-term validation |

---

# 8. What Not to Do

Запрещённые ошибки:

```text
1. Сразу объявить новую дисциплину без практических кейсов.
2. Свести IPaC к Context Engineering или Docs-as-Code.
3. Объявить SaaS-продукт доказательством теории.
4. Подменить field validation красивым vision paper.
5. Подменить methodology набором scripts.
6. Подменить Engineering of Information Production software engineering rhetoric.
7. Игнорировать уроки провалов enterprise AI adoption.
```

---

# 9. What to Do

Правильная траектория:

```text
1. Строить прикладную систему.
2. Эксплуатировать её.
3. Наблюдать failure modes.
4. Фиксировать feedback.
5. Развивать framework.
6. Проводить boundary research.
7. Вести claims ledger.
8. Отделять vision от validation.
9. Писать vision paper как стратегическую постановку.
10. Двигаться к теоретическому обобщению только на основе практики.
```

# 10. Relation to DR-002

DR-002 показал:

```text
Context Engineering — это важный AI-era practice cluster,
но его охват уже, чем заявленный IPaC knowledge lifecycle.
```

Отсюда:

```text
Context Engineering может быть полезным слоем в IPaC,
но не закрывает всю программу IPaC.
```

Этот вывод усиливает стратегическую лестницу:

```text
IPaC minimum can be a working applied system.

IPaC framework can integrate context engineering as a layer.

IPaC maximum may later generalize into Engineering of Information Production.
```
