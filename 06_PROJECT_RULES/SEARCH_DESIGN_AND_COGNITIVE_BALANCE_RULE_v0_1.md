# 1. Назначение

Этот документ фиксирует правило, возникшее в ходе практической работы IPaC над фактографическим слоем, ежедневным закрытием дня, Git/Obsidian-процессом и анализом когнитивной дельты.

Ключевой сдвиг:

```text
IPaC должен фиксировать не только то, что было произведено,
но и то, как изменилась система понимания проекта.
```

Поэтому ежедневное закрытие дня должно включать два баланса:

```text
1. Материальный баланс.
2. Когнитивный баланс.
```

---

# 2. Принцип поискового проектирования

```text
Творческая группа IPaC работает на принципах поискового проектирования.
```

Это означает:

```text
Мы не предполагаем, что конечная архитектура известна заранее.
Мы не предполагаем, что все правила можно спроектировать кабинетно.
Мы не предполагаем, что теория полностью сформирована до начала практики.
```

IPaC развивается через цикл:

```text
Практика
→ наблюдение
→ выделение закономерности
→ формулирование правила
→ проверка правила в деятельности
→ новая практика
→ уточнение правила
```

Иначе говоря:

```text
Мы не просто внедряем готовую систему.
Мы выращиваем систему через практику.
```

# 3. Материальный баланс проекта

## 3.1 Определение

```text
Материальный баланс показывает:
что было произведено.
```

Он отвечает на вопрос:

```text
Что реально добавилось в актив проекта?
```

## 3.2 Состав материального баланса

В материальный баланс входят:

```text
новые файлы;
новые исследования;
новые отчёты;
новые review;
новые decision;
новые правила;
новые инструменты;
новые скрипты;
новые экспортированные треды;
новые артефакты Obsidian;
новые Git commits;
новые Project Resources;
новые Daily Delta packs.
```

## 3.3 Назначение

Материальный баланс фиксирует продвижение проекта в физическом и операционном смысле.

Он показывает:

```text
что было сделано;
что было создано;
что было проведено через Git;
что стало доступно для повторного использования;
что может быть проверено.
```

# 4. Когнитивный баланс проекта

## 4.1 Определение

```text
Когнитивный баланс показывает:
что нового поняла система.
```

Он отвечает на вопрос:

```text
Как изменилась модель мира проекта?
```

## 4.2 Состав когнитивного баланса

В когнитивный баланс входят:

```text
новые различения;
новые формулы;
новые правила;
новые ограничения;
новые возможности;
новые границы;
новые архитектурные наблюдения;
новые интерпретации;
новые уточнения прежних позиций;
ослабленные claims;
усиленные claims;
выявленные ошибки;
обнаруженные системные риски;
переосмысленные решения.
```

## 4.3 Назначение

Когнитивный баланс показывает не просто активность проекта, а изменение качества понимания.

Он фиксирует:

```text
каким проект стал после прожитого дня;
какие правила были выделены из потока;
какие прежние представления были уточнены;
какие новые ограничения стали видны;
какие новые возможности были открыты;
какие ошибки были осознаны;
какие будущие действия стали возможны.
```

# 5. Когнитивная мнимая часть баланса

## 5.1 Определение

```text
Когнитивная мнимая часть баланса —
это слой изменений модели мира проекта,
который не сводится к физически созданным артефактам.
```

Материальный баланс показывает:

```text
что было произведено.
```

Когнитивная мнимая часть баланса показывает:

```text
как изменилась система понимания.
```

## 5.2 Почему “мнимая”

Она называется “мнимой” не потому, что нереальна.

Она мнимая в том смысле, что её нельзя напрямую измерить количеством файлов, строк, коммитов или страниц.

Она проявляется через:

```text
новые принципы;
новые правила;
новые связи;
новые формулы;
новые способы смотреть на материал;
новые режимы управления знанием.
```

## 5.3 Пример

Материальный баланс дня может сказать:

```text
Создано 2 правила.
Проведено 3 commit.
Выгружен 1 тред.
Создан 1 Daily Trace.
```

Когнитивный баланс того же дня может сказать:

```text
Понято, что фактография сохраняет право на будущий пересмотр.
Понято, что Git Diff показывает физическую дельту, а IPaC Review выделяет когнитивную дельту.
Понято, что Daily Trace должен закрываться двумя балансами.
Понято, что творческая группа работает на принципах поискового проектирования.
```

# 6. Лог когнитивных восхождений

## 6.1 Определение

```text
Лог когнитивных восхождений фиксирует переходы качества понимания,
а не только факты деятельности.
```

Обычный лог фиксирует:

```text
что было сделано.
```

Лог когнитивных восхождений фиксирует:

```text
что стало понятно;
какое правило родилось;
какая граница уточнилась;
какая ошибка была осознана;
какая новая возможность открылась;
как изменилась способность системы удерживать знание.
```

## 6.2 Связь с когнитивным балансом

Когнитивный баланс дня является структурированной формой лога когнитивных восхождений.

Он отвечает на вопрос:

```text
Какие восхождения произошли сегодня?
```

# 7. Ежедневное закрытие дня

## 7.1 Обязательная структура

Каждый значимый день IPaC должен закрываться двумя разделами:

```text
Part A — Material Balance
Part B — Cognitive Balance
```

## 7.2 Part A — Material Balance

Раздел отвечает на вопрос:

```text
Что было создано?
```

Примерная структура:

```markdown
# Part A — Material Balance

## Files created

## Reports / Reviews / Decisions

## Rules created

## Thread exports

## Git commits

## Project sync actions

## Open material debts
```

## 7.3 Part B — Cognitive Balance

Раздел отвечает на вопрос:

```text
Что было понято?
```

Примерная структура:

```markdown
# Part B — Cognitive Balance

## New distinctions

## New rules extracted from the flow

## New formulas

## Changed interpretations

## Weakened / strengthened claims

## New risks

## New opportunities

## Cognitive ascent events

## Open cognitive debts
```

# 8. Связь с Factography Layer

Фактографический слой отвечает на вопрос:

```text
Что реально было?
```

Материальный баланс отвечает на вопрос:

```text
Что было произведено?
```

Когнитивный баланс отвечает на вопрос:

```text
Что было понято?
```

Их нельзя подменять друг другом.

```text
Factography
→ Material Balance
→ Cognitive Balance
→ Daily Trace
→ Project Delta
→ Canon / Future Reinterpretation
```

---

# 9. Связь с Git Diff

Git Diff показывает физическую дельту:

```text
какие строки, файлы, изображения и артефакты изменились.
```

Когнитивный баланс показывает смысловую дельту:

```text
какие изменения понимания возникли на основе этой физической дельты.
```

Рабочая формула:

```text
Git Diff = физическая дельта.
Cognitive Balance = смысловая дельта.
```

---

# 10. Связь с Project Trace

Project Trace должен получать не только список созданных файлов, но и когнитивный итог дня.

В Project Trace должны попадать:

```text
ключевые материальные результаты;
ключевые когнитивные восхождения;
новые правила;
изменение текущей позиции проекта;
открытые долги;
следующий фокус.
```

---

# 11. Связь с поисковым проектированием

Поисковое проектирование требует регулярного выведения когнитивного баланса.

Почему:

```text
Если мы не фиксируем, что поняли,
то мы не видим, как развивается проект.
```

Иначе проект может иметь много файлов и коммитов, но не иметь видимой траектории развития понимания.

Поэтому:

```text
Материальный баланс без когнитивного баланса показывает активность,
но не показывает развитие.
```
