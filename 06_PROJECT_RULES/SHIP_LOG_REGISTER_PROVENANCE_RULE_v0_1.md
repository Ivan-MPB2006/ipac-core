# 1. Назначение

Это правило фиксирует строгую трассируемость происхождения записей в IPaC.

Главная формула:

```text
Бортовой журнал может иметь записи без разнесения в Реестры.
Но Реестры не могут иметь записей без ссылки на Бортовой журнал.
```

Бортовой журнал — первичный слой фиксации события.  
Реестры / Регистры — производные специализированные контуры.

В специализированном Реестре не должно быть записей “с воздуха”.

---

# 2. Базовый принцип provenance

```text
Любая запись в специализированном Реестре должна иметь происхождение.
```

Минимальная цепочка происхождения:

```text
Фактография
→ Бортовой журнал
→ Спектральный разбор
→ Разнесение
→ Реестр / Артефакт
→ Git-проводка
→ Обратная трассировка
```

Если происхождение не указано, запись считается неполной.

---

# 3. Различие Бортового журнала и Реестров

## 3.1 Бортовой журнал

Бортовой журнал фиксирует ход дня и значимые события.

Он может содержать:

```text
сырые события;
незрелые наблюдения;
точки перегиба;
подозрения на будущие правила;
открытые сигналы;
события без маршрутизации;
пустые записи без продолжения;
кандидаты на будущий разбор агентом.
```

Это нормально.

```text
Бортовой журнал допускает незрелые записи.
```

## 3.2 Реестры / Регистры

Реестры являются специализированными производными контурами.

Они могут содержать:

```text
правила;
решения;
гипотезы;
claims;
онтологические элементы;
архитектурные инварианты;
исследовательские долги;
MVP-задачи;
Codex/MCP-задачи;
Project Trace entries.
```

Реестр требует provenance.

```text
Запись в Реестре без ссылки на Бортовой журнал запрещена.
```

# 4. Номер записи и номер проводки

## 4.1 Ship Log ID

Каждая значимая запись Бортового журнала должна иметь идентификатор.

Рекомендуемый формат:

```text
LOG-YYYY-MM-DD-NNN
```

Пример:

```text
LOG-2026-06-17-004
```

## 4.2 Git Transaction ID

Git commit hash является естественным номером проводки.

Он может использоваться как:

```text
Git Transaction ID
Номер проводки
Provenance ID
```

Пример:

```text
Git Transaction: db9f4f9
```

Полная фиксация события возникает тогда, когда:

```text
Бортовая запись существует
+
производные артефакты созданы
+
Git commit зафиксировал пакет изменений
```

# 5. Строгость “до блокчейна”

IPaC не обязан быть блокчейном технически.

Но он должен заимствовать принцип доказательной сцепки:

```text
нет записи без происхождения;
нет производного артефакта без источника;
нет принятого изменения без проводки;
нет проводки без понятного смыслового пакета.
```

Git обеспечивает практический антиэнтропийный храповик.

```text
Commit hash = доказательство фиксации состояния.
```

---

# 6. Сгусток и спектральное разнесение

Событие дня часто рождается как смысловой сгусток.

```text
Сгусток = контекстно запитанное событие, мысль или перегиб,
который ещё не разложен на специализированные составляющие.
```

Один сгусток может содержать несколько гармоник:

```text
rule_candidate;
ontology_note;
project_trace;
decision_candidate;
research_backlog;
codex_task;
mcp_spec;
cognitive_ascent;
canon_candidate;
parking_lot_item.
```

Поэтому нельзя преждевременно класть сгусток только в один Реестр.

Правильный порядок:

```text
1. Зафиксировать сгусток в Бортовом журнале.
2. Провести спектральный разбор.
3. Определить гармоники.
4. Разнести по соответствующим Реестрам / артефактам.
5. Провести групповым Git commit.
6. В производных записях указать источник.
```

---

# 7. Бортовая запись без продолжения

В Бортовом журнале допустима запись без последующего разнесения.

Статусы:

```text
captured_only;
immature_signal;
awaiting_pattern;
parking_signal;
agent_review_later;
not_yet_routed.
```

Такая запись не является ошибкой.

Она означает:

```text
сигнал замечен,
но ещё не созрел для специализированного Реестра.
```

Будущий агент может пройти по Бортовому журналу, собрать статистику повторений и предложить:

```text
это созрело для правила;
это созрело для канона;
это паттерн;
это НКУ;
это ложный след;
это стоит оставить в parking.
```

---

# 8. НКУ — новый качественный уровень

```text
НКУ — новый качественный уровень.
```

Это момент, когда набор событий, повторений, точек перегиба и разнесённых сгустков образует устойчивую новую способность системы.

Иначе:

```text
НКУ = восхождение / набор когнитивной высоты.
```

Признаки НКУ:

```text
1. Появился повторяемый паттерн.
2. Появилось новое правило.
3. Появился новый контур управления.
4. Изменилась модель мира проекта.
5. Появилась новая операционная способность.
6. Изменился порог ординарности.
7. Система начала делать то, что раньше требовало ручного напряжения.
```

НКУ может быть обнаружен позже при проходе по Бортовому журналу:

```text
агент видит несколько похожих сигналов;
сравнивает их;
выделяет устойчивую гармонику;
предлагает НКУ-кандидат.
```

# 10. Формат записи в специализированном Реестре

Каждая запись в Реестре должна ссылаться на Бортовой журнал.

```markdown

```yaml
entry_type: <rule / decision / ontology_note / claim / backlog / task>

source_factography: <thread_export / file / screenshot / command>
git_transaction: <commit-hash or pending>

spectral_component: <rule_candidate / ontology_note / project_trace / ...>
status: candidate / accepted / committed / parked
```

### Derived meaning / Производный смысл

### Evidence / Основание

### Links / Связи
```

# 11. Групповая Git-проводка

Когда один сгусток разнесён по нескольким артефактам, предпочтительна групповая Git-проводка.

Пример:

```text
LOG-2026-06-17-004 породил:
- правило;
- запись в Project Trace;
- обновление Daily Register;
- ontology note.
```

Тогда commit может быть один:

```powershell
git add <rule-file>
git add <daily-register>
git add <ontology-note>
git add <project-trace>

git commit -m "trace: route ship log event LOG-2026-06-17-004"
```

Смысл:

```text
один смысловой сгусток
→ несколько спектральных составляющих
→ одна групповая проводка.
```

# 12. Слой для человека

Человек должен помнить:

```text
Сначала Бортовой журнал.
Потом маршрутизация.
Потом Реестры.
Потом Git.
```

Не надо сразу решать, “куда навсегда положить” живую мысль.

Сначала её надо безопасно зафиксировать.

# 13. Слой для ИИ

ИИ должен помогать:

```text
1. Выделять события, достойные Бортового журнала.
2. Предлагать Ship Log ID.
3. Определять event characteristic.
4. Раскладывать сгусток на спектральные составляющие.
5. Предлагать маршруты разнесения.
6. Проверять, что каждая запись в Реестре имеет source_ship_log_entry.
7. Предлагать group commit.
8. Накапливать статистику незрелых сигналов.
9. Предлагать НКУ-кандидаты.
```

ИИ не должен создавать запись в специализированном Реестре без ссылки на Бортовой журнал или фактографию.

# 14. Слой для Obsidian

Obsidian должен обеспечивать:

```text
онтологию;
семантику;
топологию;
типологику;
связность;
валентность;
стекИруемость.
```

Для этого:

```text
Бортовой журнал содержит первичные события.
Реестры содержат специализированные производные.
Links связывают LOG → REG → Artifact → Commit.
Tags / properties фиксируют типы и характеристики.
Backlinks показывают происхождение смыслов.
```

# 15. Анти-паттерны

## 15.1 Запись с воздуха

```text
Запись в Реестре без source_ship_log_entry.
```

Запрещено.

## 15.2 Преждевременное разнесение

```text
Сгусток положен только в один Реестр,
хотя содержит несколько гармоник.
```

Ошибка: теряется спектр.

## 15.3 Потеря контекстного импульса

```text
Правило есть, но непонятно, из какого события оно родилось.
```

Ошибка: исчезает происхождение.

## 15.4 Коммит без смыслового пакета

```text
Git commit фиксирует случайную смесь файлов без общей смысловой причины.
```

Ошибка: нарушается трасса.
