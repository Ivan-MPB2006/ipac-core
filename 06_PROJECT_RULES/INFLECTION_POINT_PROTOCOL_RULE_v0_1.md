---

# 1. Назначение

Это правило фиксирует принцип протоколирования **точек перегиба** в IPaC.

Главная формула:

```text
Не суетиться.
Не мельчить.
Но и не пропускать важное.
```

IPaC не должен превращаться в бюрократию, где протоколируется каждая мелочь.  
Но IPaC также не должен пропускать моменты, где меняется направление развития системы.

---

# 2. Базовый принцип

```text
Протоколировать не всё подряд.
Протоколировать точки перегиба.
```

Точка перегиба — это момент, в котором изменяется:

```text
направление движения;
уровень работы;
роль инструмента;
контур управления;
статус артефакта;
режим мышления;
архитектурная гипотеза;
производственный процесс;
способ фиксации знания;
способ принятия решений.
```

---

# 3. Почему это важно

Сложные операции нельзя понимать только по итоговому результату.

Важны места, где система:

```text
сменила направление;
перешла на новый уровень;
обнаружила ограничение;
изменила способ работы;
ввела новое правило;
выделила новый контур;
приняла новую архитектурную рамку.
```

Именно в таких точках рождается новое качество системы.

---

# 4. Дифференцирование и интеграция

Ключевая методологическая формула:

```text
Интегрируют только после дифференцирования.
Дифференцирование невозможно без фактографии.
Фактография особенно нужна в точках перегиба.
```

Без фактографической подложки невозможно честно понять:

```text
где началось новое направление;
почему изменилась траектория;
какие данные повлияли на пересмотр;
что именно было принято;
что было отвергнуто;
какая новая способность возникла.
```

---

# 5. Что считать точкой перегиба

Точкой перегиба считается событие, после которого система уже не вполне та же самая, что до него.

Примеры:

```text
1. Обнаружено, что модель не является архивом.
2. Принято решение вести Daily Register.
3. Получен рабочий экспорт треда в Markdown с графикой.
4. Введён Factography Layer.
5. Введён Material / Cognitive Balance.
6. Создан staging-пакет фактографии дня.
7. Закрыт операционный день.
8. Принято решение переносить интеллект в Project OS.
9. Принято решение о федерации Project-контуров.
10. НИР / НИОКР / MVP вынесены из управляющего Project.
```

---

# 6. Что НЕ является точкой перегиба

Не надо протоколировать как точку перегиба:

```text
каждую мелкую команду PowerShell;
каждый просмотр файла;
каждую техническую ошибку без урока;
каждую промежуточную формулировку;
каждый обычный git status;
каждое повторение уже принятого правила.
```

Но техническая ошибка становится точкой перегиба, если она привела к:

```text
новому правилу;
новому ограничению;
новой процедуре;
новому способу защиты процесса;
изменению архитектуры.
```

---

# 7. Формат записи точки перегиба

Каждая точка перегиба должна фиксироваться в Daily Register или Daily Trace в формате:

```markdown
- [ ] YYYY-MM-DD HH:mm — <краткое название точки перегиба>
      Type: inflection_point / cognitive_system_event / operational_governance_event
      Status: detected / accepted / committed / parked
      What changed: <что изменилось>
      Why it is inflection: <почему это точка перегиба>
      Affected layer: <factography / trace / project_os / NIR / NIOKR / MVP / canon / tooling>
      Resulting action: <какое действие или правило возникло>
      Include in: DAILY_TRACE_YYYY-MM-DD
```

---

# 8. Связь с Daily Register

Daily Register не должен быть мусорным журналом всего подряд.

Его роль:

```text
держать точки перегиба;
держать важные артефакты;
держать решения;
держать долги;
держать события, которые обязательно должны войти в Daily Trace.
```

То есть Daily Register должен быть:

```text
операционным буфером важного,
а не стенограммой всего.
```

Полная стенограмма / первичный поток живёт в Factography Layer через thread export.

---

# 9. Связь с Factography Layer

Фактография отвечает:

```text
что реально было.
```

Inflection Point Protocol отвечает:

```text
где в потоке произошёл перелом траектории.
```

То есть:

```text
Factography сохраняет весь поток.
Inflection Protocol ставит маркеры смысла на ключевых переломах.
```

---

# 10. Связь с Cognitive Balance

Когнитивный баланс дня должен включать все значимые точки перегиба.

Потому что именно они показывают:

```text
как изменилась система понимания проекта;
какое новое правило выделено из практики;
какой новый контур стал видимым;
какое прежнее представление было пересмотрено.
```

Точка перегиба — это кандидат в запись лога когнитивных восхождений.

---

# 11. Практическое правило для ассистента

Ассистент должен помогать различать:

```text
шум;
операционное событие;
значимое событие;
точку перегиба.
```

Если событие является точкой перегиба, ассистент должен предложить:

```text
1. короткую формулировку;
2. запись в Daily Register;
3. возможный будущий артефакт;
4. связь с текущими правилами;
5. необходимость Git-проводки, если возник новый артефакт.
```

---

# 12. Практическое правило для человека

Человек не обязан протоколировать всё.

Но человек должен остановиться и зафиксировать момент, если чувствует:

```text
мы изменили направление;
мы поднялись уровнем выше;
мы увидели новый контур;
мы нашли фундамент;
мы изменили способ работы;
мы поняли, что старый способ больше не годится;
мы начали новый режим деятельности.
```

Рабочая команда:

```text
В реестр дня: это точка перегиба — <описание>
```

```text

```

```text
Если всё объявлять важным, важное перестаёт быть видимым.
```

## 13.3 Пропуск перегиба

```text
Если точка перегиба не зафиксирована,
позже изменение будет выглядеть как случайная смена мнения.
```

## 13.4 Ретроспективная подмена

```text
Нельзя задним числом представлять новый уровень понимания как изначально очевидный.
```
