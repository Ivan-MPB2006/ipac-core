- источником базовых принципов;
- верхним ориентиром для канона, регистров, research, Obsidian, Git, Codex/MCP и будущих агентов.

```text
active boot-level constitution
```

Это не final canon.  
Это рабочая Конституция v0.1, подлежащая controlled rebuild через trace, review и decision.

# 1. Главная проблема

Современный ИИ сделал производство output дешёвым.

```text
AI made output cheap.
Now meaning is the bottleneck.
```

Перевод:

```text
ИИ сделал производство output дешёвым.
Теперь узким местом стал смысл.
```

ИИ способен быстро генерировать тексты, планы, схемы, summaries, код и варианты решений.

Но главным узким местом остаётся не output, а смысл:

- удержание замысла;
- сохранение контекста;
- различение фактов, гипотез и интерпретаций;
- прослеживаемость решений;
- проверка оснований;
- работа с неопределённостью;
- управляемая пересборка знания;
- превращение живого смысла в переносимое и проверяемое знание.

# 2. Предмет IPaC

IPaC работает не просто с документами, заметками, промптами или кодом.

Предмет IPaC:

```text
живая знаниевая и смысловая среда
```

IPaC работает с:

- живым смыслом;
- знаниевыми сгустками;
- артефактами;
- каноном;
- регистрами;
- trace;
- исследовательскими пробелами;
- обратной связью;
- решениями;
- управляемой пересборкой;
- деятельностным переходом к исполнению.

# 3. Главная задача IPaC

Главная задача IPaC — снижать сложность работы с живым знанием и смыслом на порядки.

Аналогия:

```text
Операционные системы снизили сложность работы с вычислительными средами.
IPaC должен снизить сложность работы со знаниевыми и смысловыми средами.
```

Операционная система сделала вычислительный процесс доступным не только уникальным специалистам, глубоко понимающим машину, но значительно более широкому кругу профессионалов.

IPaC должен сделать работу со сложным живым знанием доступной не только уникальным архитекторам, методологам и исследователям, но более широкому кругу специалистов, способных работать с большими смысловыми системами при поддержке AI.

---

# 4. Смена предмета сложности

Классическая операционная система работает со сложностью вычислительной среды:

- процессор;
- память;
- ввод / вывод;
- файловая система;
- задачи;
- процессы;
- очереди;
- ошибки;
- восстановление;
- права доступа;
- ресурсы.

IPaC работает с другой сложностью:

- смысловой;
- контекстной;
- интерпретационной;
- исследовательской;
- коммуникативной;
- деятельностной;
- валидационной;
- архитектурной;
- организационной.

Предмет другой.  
Сложность другого уровня, плана и характера.

---

# 5. Основная формула IPaC

```text
IPaC — это операционный контур,
который снижает сложность работы со смыслом,
как операционные системы снизили сложность работы с вычислениями.
```

Короткая формула:

```text
OS делает вычисления управляемыми.
IPaC делает живое знание управляемым.
```

# 6. Базовый контур IPaC

IPaC разворачивается как жизненный цикл живого знания:

```text
Living Meaning
→ Knowledge Clump
→ Knowledge Artifact
→ Canon
→ Trace
→ Feedback / Evidence
→ Review
→ Decision
→ Controlled Rebuild
→ Updated Canon
```

Перевод:

```text
Живой смысл
→ Знаниевый сгусток
→ Знаниевый артефакт
→ Канон
→ След / трасса
→ Обратная связь / свидетельства
→ Разбор / ревью
→ Решение
→ Управляемая пересборка
→ Обновлённый канон
```

# 7. Роли компонентов полного контура

## Chat

Чат — это не хранилище и не source-of-truth.

Чат — место высечения искры нового знания.

Он нужен для:

- мозгового штурма;
- когнитивного трения;
- появления новых связей;
- первичного смыслового резонанса;
- прояснения вопросов;
- запуска нового знания.

Но чат не должен становиться долговременной памятью проекта.

## ChatGPT Project

Project удерживает рабочую рамку.

Он нужен для:

- reasoning;
- review;
- восстановления канона;
- контроля guardrails;
- работы с source-of-truth;
- управления фазами и gates.

## Obsidian

Obsidian — пространственная инженерная карта.

Он нужен для:

- объёмного восприятия;
- навигации;
- чтения;
- связей;
- канона;
- регистров;
- parking lot;
- trace;
- работы человека с большим смысловым полем.

```text
Чат — щель восприятия.
Obsidian — cockpit.
```

## Git

Git — след, история, версия, откат.

Он нужен для:

- фиксации изменений;
- управления версиями;
- контроля канона;
- trace;
- rollback;
- инженерного отношения к знанию.

## GitHub

GitHub — внешний удалённый контур хранения и передачи.

Он нужен для:

- backup;
- remote trace;
- переносимости;
- будущей совместной работы;
- подготовки к Codex / agents / review workflows.

## Deep Research

Deep Research — разведка внешнего поля.

Он нужен для:

- закрытия research gaps;
- проверки границ IPaC;
- сравнения с существующими дисциплинами;
- выявления source limitations;
- укрепления или пересмотра гипотез.

Deep Research не доказывает IPaC автоматически.  
Он даёт research support, но не field validation.

## Codex / MCP

Codex + MCP — деятельностно-силовой слой IPaC.

Это не просто “исполнитель кода”.

Это компонент, который должен уметь:

- создавать директории;
- переносить информационные массивы;
- раскладывать файлы;
- обновлять Obsidian Vault;
- проверять структуры;
- собирать ZIP-пакеты;
- работать с Git;
- материализовывать артефакты;
- автоматизировать рутину;
- снижать барьеры между замыслом и действием.

```text
Project думает.
Obsidian показывает.
Git фиксирует.
Codex двигает.
MCP даёт руки к миру.
```

# 8. Полный технологический контур

Целевой полный цикл IPaC:

```text
искра смысла в чате
↓
рабочая рамка в Project
↓
объёмная картина в Obsidian
↓
след и версии в Git
↓
внешняя проверка через Deep Research
↓
деятельностный слой через Codex / MCP
↓
коммуникация и полевая обратная связь
↓
review / decision / controlled rebuild
↓
обновлённый canon
```

Цель этого контура:

```text
скорость и лёгкость потока
```

Не хаотическая скорость.  
А управляемый поток:

- со следом;
- с проверкой;
- с обратимостью;
- с каноном;
- с открытыми вопросами;
- с parking lot;
- с возможностью пересборки.

---

# 9. Главный принцип сложности

Сложность нельзя уничтожить.

Её можно:

- вынести в правильные слои;
- скрыть за интерфейсом;
- разложить по артефактам;
- связать trace;
- отдать исполнение силовому контуру;
- оставить человеку смысловое суждение;
- удерживать через канон, регистры и review.

IPaC не упрощает смысл до мёртвого документа.  
IPaC создаёт операционный контур, в котором живой смысл может сохраняться, двигаться, проверяться и пересобираться.

# 10. Что IPaC не является

IPaC — это не:

- обычный knowledge management;
- обычный project management;
- просто Obsidian Vault;
- просто Git-репозиторий;
- просто AI workflow;
- просто Docs as Code;
- просто RAG;
- просто Codex automation;
- просто MVP-machine;
- просто content factory.

IPaC использует эти элементы, но не сводится к ним.

# 12. Нормативные запреты

Нельзя:

- превращать чат в source-of-truth;
- считать красивую сводку доказательством;
- смешивать Need, Concept, Architecture, Hypothesis, Research, MVP и Execution;
- переносить execution candidates в architecture core;
- считать research support field validation;
- считать accepted-candidate final canon;
- подключать Codex без task pack;
- запускать Deep Research без конкретного research gap;
- использовать Obsidian как самоцель;
- превращать Git в формальность;
- принимать raw feedback как решение без review;
- обновлять canon без trace.

---

# 13. Нормативные обязанности

Нужно:

- сохранять живой смысл через артефакты;
- фиксировать source-of-truth;
- держать open questions видимыми;
- поддерживать parking lot;
- вести trace;
- разделять слои;
- проверять гипотезы;
- фиксировать validation level;
- использовать Git для истории изменений;
- использовать Obsidian для объёмного восприятия;
- подключать Codex/MCP только к ограниченным деятельностным задачам;
- пропускать изменения канона через review / decision / controlled rebuild.
